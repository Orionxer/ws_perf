# PRD: WebSocket 性能与上传链路演示项目

**文档版本：** v1.0  
**创建日期：** 2026-03-19  
**项目名称：** `ws_perf`  
**文档用途：** 记录本项目的产品目标、范围、关键流程、验收标准与限制，供后续复盘、迭代和交接使用。

---

## 1. 背景

当前项目用于验证一个最小闭环：

- 服务端能够同时承载 HTTP 静态服务与 WebSocket 通信
- 浏览器端可以作为“监控面板”实时观察连接状态
- Node.js 客户端可以在收到指令后上传视频文件
- 服务端可以保存文件并输出基础传输指标

该项目不是完整业务产品，而是一个偏工程验证和演示性质的原型系统。它的价值主要体现在以下几个方面：

- 快速验证 WebSocket 指令下发与二进制传输链路
- 提供可视化监控页面，方便联调和演示
- 为后续扩展成正式传输平台、设备控制面板或性能压测工具提供基础样板

---

## 2. 产品目标

### 2.1 核心目标

构建一个可运行、可观察、可验证的 WebSocket 文件上传演示系统，完成“监控端发起指令 -> 客户端上传视频 -> 服务端保存并反馈结果”的完整闭环。

### 2.2 具体目标

- 支持客户端接入与在线状态管理
- 支持监控端查看在线客户端列表
- 支持监控端选择单个客户端并触发视频上传
- 支持客户端以流式方式上传视频文件
- 支持服务端保存上传结果并计算 SHA256、RTT、上传耗时、带宽
- 支持监控端查看上传进度和结果
- 支持在监控端直接播放服务端已保存的视频文件
- 支持基础连接治理能力，例如心跳检测和关闭所有客户端

### 2.3 非目标

以下内容不在当前版本范围内：

- 多文件管理与历史版本管理
- 用户系统、权限体系和登录认证
- 断点续传
- 并发多任务上传调度
- 数据库存储
- 文件元数据持久化
- 生产级安全控制、审计和告警
- 正式压测报告与系统级性能基线

## 3. 用户与使用场景

### 3.1 目标用户

- 开发人员：用于本地联调 WebSocket 与上传流程
- 测试人员：用于验证连接状态、上传行为和结果展示
- 演示人员：用于对外或对内展示上传链路能力

### 3.2 核心场景

#### 场景 A：本地联调

开发者启动服务端、监控面板和 demo 客户端，通过监控页触发上传，观察日志、进度和最终结果。

#### 场景 B：跨设备访问

开发者在电脑上运行服务端，手机通过局域网访问监控页面，实时查看客户端连接和上传状态。

#### 场景 C：链路验证

团队在需求评审或技术复盘时，通过本项目快速说明：

- WebSocket 指令如何下发
- 二进制数据如何上传
- 服务端如何汇总结果
- 前端如何展示上传状态

## 5. 信息架构与模块划分

### 5.1 服务端

职责：

- 提供 `dashboard/dist` 静态页面
- 提供 `/resource/*` 文件访问
- 提供 WebSocket 服务
- 区分 monitor 与普通 client
- 向客户端下发上传命令
- 接收上传文件并计算结果
- 将连接状态、消息和结果广播给监控端

### 5.2 监控面板

职责：

- 显示在线客户端列表
- 显示客户端详情页
- 展示消息日志
- 发起上传命令
- 展示上传状态、上传结果和视频播放器
- 切换心跳检测
- 关闭所有客户端连接

### 5.3 演示客户端

职责：

- 连接 WebSocket 服务端
- 接收 `/upload_video` 指令
- 发送 `/file_start`
- 读取本地视频并流式上传
- 发送 `/file_end`
- 接收 `/file_received` 并输出本地统计结果

## 6. 核心流程

### 6.1 客户端接入流程

1. 服务端启动，监听 `8080`
2. demo 客户端通过 WebSocket 接入
3. 服务端为客户端分配唯一 `clientId`
4. 服务端向所有 monitor 广播 `[CONNECT]`
5. 监控面板刷新在线客户端列表

### 6.2 上传流程

1. monitor 在客户端详情页点击“上传视频”
2. monitor 向服务端发送 `[COMMAND] UPLOAD_VIDEO <clientId>`
3. 服务端清理旧的 `resource/starship.mp4`
4. 服务端向目标客户端发送 `/upload_video`
5. 客户端收到命令后发送 `/file_start`
6. 客户端按 chunk 发送视频二进制数据
7. 服务端累计接收字节数并向 monitor 广播进度
8. 客户端发送 `/file_end`
9. 服务端拼接文件、保存文件、计算 SHA256 与带宽信息
10. 服务端向客户端发送 `/file_received`
11. 服务端向 monitor 广播上传成功结果
12. monitor 展示结果，并尝试播放新视频

### 6.3 心跳检测流程

1. 服务端每 10 秒向 monitor 和 client 发送 ping
2. 连接若未返回 pong，则判定失活
3. 服务端移除该连接并向 monitor 广播 `[DISCONNECT]`
4. monitor 页面刷新在线状态

## 7. 通信协议

### 7.1 Monitor -> Server

```text
[COMMAND] UPLOAD_VIDEO <clientId>
[COMMAND] TOGGLE_HEARTBEAT
[COMMAND] CLOSE_ALL_CLIENTS
```

### 7.2 Server -> Client

```text
/upload_video
/file_received
```

### 7.3 Client -> Server

```text
/file_start
<binary chunks>
/file_end
```

### 7.4 Server -> Monitor

```text
[CONNECT] <clientId>-<ip>
[DISCONNECT] <clientId>-<ip>
[MESSAGE] <clientId>-<ip>: <message>
[COMMAND_RESULT] UPLOAD_VIDEO_REQUESTED <clientId>
[COMMAND_RESULT] UPLOAD_VIDEO_PROGRESS <clientId> <bytes> <mb>
[COMMAND_RESULT] UPLOAD_VIDEO_SAVED <clientId> <bytes> <mb> <sha256> <rttMs> <bandwidthMBps> <bandwidthMbps> <uploadTimeMs>
[COMMAND_RESULT] UPLOAD_VIDEO_FAILED <clientId> <reason>
[COMMAND_RESULT] CLOSE_ALL_CLIENTS <count>
[COMMAND_RESULT] HEARTBEAT_ENABLED
[COMMAND_RESULT] HEARTBEAT_DISABLED
[SYSTEM] HEARTBEAT ON|OFF
```

## 8. 功能需求

### FR-1 在线客户端展示

系统应能够在监控页面实时展示所有已连接客户端。

验收标准：

- 新客户端接入后，列表自动出现该客户端
- 客户端断开后，列表自动移除
- 列表项至少包含 `clientId`、IP、连接时间

### FR-2 客户端详情查看

系统应支持查看单个客户端详情。

验收标准：

- 可从列表跳转到详情页
- 详情页可查看该客户端收到的消息记录
- 若客户端已断开，详情页自动返回列表页

### FR-3 触发视频上传

系统应支持从监控页面向指定客户端下发上传视频命令。

验收标准：

- 点击上传按钮后，服务端成功向目标客户端发送 `/upload_video`
- 若客户端不存在，页面显示失败信息
- 上传期间按钮应处于禁用或忙碌状态

### FR-4 流式上传视频

客户端应在收到上传命令后，流式发送本地视频文件。

验收标准：

- 客户端先发送 `/file_start`
- 客户端随后发送二进制 chunk
- 结束时发送 `/file_end`
- 服务端能完整接收并保存文件

### FR-5 上传进度展示

监控页面应展示当前上传进度。

验收标准：

- 上传开始时页面出现“上传中”状态
- 接收 chunk 过程中页面显示累计字节数与 MB 值
- 上传结束后页面切换为成功或失败状态

### FR-6 上传结果统计

服务端应在上传完成后生成结果统计。

验收标准：

- 计算并展示文件大小
- 计算并展示 SHA256
- 计算并展示 RTT
- 计算并展示上传耗时
- 计算并展示带宽 MB/s 和 Mbps

### FR-7 视频回放

监控页面应支持播放服务端当前保存的视频文件。

验收标准：

- 页面会通过 `HEAD /resource/starship.mp4` 检查文件是否存在
- 文件存在时展示播放器
- 播放器可通过 `Range` 请求加载视频内容

### FR-8 心跳检测开关

监控页面应支持切换心跳检测开关。

验收标准：

- 点击按钮后服务端切换心跳状态
- 页面上能够反映当前心跳状态
- 关闭心跳时服务端不主动清理超时连接

### FR-9 一键关闭所有客户端

监控页面应支持关闭所有客户端连接。

验收标准：

- 点击按钮后服务端遍历所有 client 连接并关闭
- monitor 仍保持可用
- 页面上的客户端数量清零

- 支持至少一个客户端完成 12MB 量级视频文件上传

### NFR-4 兼容性

- 支持 Node.js 环境运行服务端与 demo 客户端

## 11. 验收标准

本项目在以下条件满足时，可视为当前版本验收通过：

1. 服务端启动后可访问 `http://localhost:8080`
2. 监控页面能正确显示在线客户端
3. demo 客户端连接后能在页面中出现
4. 点击上传后，客户端能收到 `/upload_video`
5. 服务端能接收到完整视频并保存到 `resource/starship.mp4`
6. 监控页面能显示上传进度与最终结果
7. 页面能播放上传后的视频文件
8. 心跳切换和关闭所有客户端功能可用

## 12. 已知限制

- 文件名当前固定为 `starship.mp4`
- 服务端只保留一份最新上传结果，不保留历史版本
- 上传前会删除已有目标文件
- demo 客户端目标地址默认写死为 `ws://localhost:8080`
- monitor 与 client 的识别方式依赖请求头推断，存在环境耦合
- 当前没有鉴权机制，任何可接入 monitor 的页面都能下发命令
- 当前无自动化测试覆盖

## 13. 风险与问题

### 13.1 技术风险

- 大文件上传依赖内存拼接 `Buffer.concat`，文件继续增大时会有内存压力
- 缺少断线恢复与失败重试机制
- monitor/client 识别策略在特殊浏览器或代理场景下可能误判
- 静态资源托管使用 `dashboard/dist`，如果源码变更后未重新构建，页面与源码会不一致

### 13.2 交付风险

- 当前项目偏原型，若直接用于正式业务场景会缺少权限、安全和可运维能力
- 当前统计结果适合演示，不等于严格性能压测基线

## 14. 复盘建议

下次复盘建议重点关注以下问题：

- 当前协议设计是否足够清晰，是否需要引入 JSON 消息结构
- 上传统计是否应由服务端与客户端分别记录并对账
- 是否需要支持多文件、多任务和历史记录
- monitor/client 连接类型识别是否需要显式握手协议替代
- 是否需要将上传过程持久化到数据库或日志系统
- 是否需要补充自动化测试、压测脚本和异常场景测试
