# WebSocket 监控面板需求文档

## 1. 项目概述

### 1.1 项目背景
现有 Node.js WebSocket 服务器（使用 `ws` 库实现，监听端口 8080），能够监听所有客户端的连接、断开以及消息发送事件。

### 1.2 项目目标
开发一个动态 WebSocket 监控面板，以卡片视图实时展示所有客户端状态，支持查看客户端详情及实时消息流。

### 1.3 设计原则
- 尽可能简单实现
- 不考虑性能优化
- 无需数据库和消息缓存
- 实时显示，无需持久化

---

## 2. 功能需求

### 2.1 客户端列表视图（首页）

#### 2.1.1 卡片展示
- 以卡片网格视图展示所有已连接的客户端
- 每个客户端卡片显示以下信息：
  - **唯一客户端 ID**：服务器随机生成的唯一标识
  - **连接时间**：客户端建立连接的时间
  - **IP 地址**：客户端的 IP 地址

#### 2.1.2 动态更新
- **新增连接**：当服务器推送 `[CONNECT]` 事件时，立即新建客户端卡片
- **断开连接**：当服务器推送 `[DISCONNECT]` 事件时，立即删除对应客户端卡片

#### 2.1.3 交互
- 点击客户端卡片，跳转到该客户端的详情页面

### 2.2 客户端详情视图

#### 2.2.1 客户端信息展示
- 显示客户端基本信息：
  - 唯一客户端 ID
  - 连接时间
  - IP 地址

#### 2.2.2 消息流展示
- 实时显示该客户端发送的所有消息
- 消息以纯文本形式展示，无需高亮、格式化或特殊标识
- 使用滚动条支持查看历史消息（在当前会话范围内）
- 新消息自动追加到列表底部

#### 2.2.3 交互
- 提供返回按钮，返回客户端列表视图

## 4. 接口规范

### 4.1 连接规范
- **WebSocket 地址**：`ws://localhost:8080`
- **浏览器识别**：服务器通过检测 WebSocket 握手请求头的 `User-Agent` 字段来识别浏览器客户端
- **监控面板标识**：浏览器客户端自动被视为监控面板，服务器向其推送所有客户端状态

### 4.2 服务器消息格式

服务器向监控面板推送纯文本消息，通过关键词识别事件类型：

#### 4.2.1 新连接事件
```
[CONNECT] <client-id>-<ip-address>
```
示例：`[CONNECT] abc123-192.168.1.100`

#### 4.2.2 断开连接事件
```
[DISCONNECT] <client-id>-<ip-address>
```
示例：`[DISCONNECT] abc123-192.168.1.100`

#### 4.2.3 消息事件
```
[MESSAGE] <client-id>-<ip-address>: <message-content>
```
示例：`[MESSAGE] abc123-192.168.1.100: Hello World`

### 4.3 初始状态
- 监控面板连接成功后，服务器立即推送当前已连接的所有客户端列表
- 每个客户端一条 `[CONNECT]` 消息
- 之后持续推送实时事件

### 4.4 消息解析规则
- 通过消息前缀识别事件类型：
  - `[CONNECT]` → 新增客户端卡片
  - `[DISCONNECT]` → 删除客户端卡片
  - `[MESSAGE]` → 追加消息到对应客户端的消息列表
- 从消息中提取 `client-id` 和 `ip-address`（以 `-` 分隔）
- 对于 `[MESSAGE]`，提取冒号后的内容作为消息文本

## 5. 页面设计

### 5.1 客户端列表页
- **布局**：响应式网格布局，卡片自动换行
- **卡片样式**：简洁的卡片设计，显示 ID、连接时间、IP
- **导航**：标题栏显示 "WebSocket 监控面板"

### 5.2 客户端详情页
- **布局**：
  - 顶部：客户端信息卡片
  - 下方：消息流区域
- **消息流**：垂直滚动列表，纯文本展示

### 5.3 样式要求
- 简洁清晰，无复杂动画
- 响应式设计，支持桌面浏览器

---

## 6. 非功能需求

### 6.1 不需要实现的功能
- ❌ 统计面板（连接数统计等）
- ❌ 排序和筛选功能
- ❌ 消息高亮和格式化
- ❌ 消息分类或标识
- ❌ 自动滚动控制
- ❌ 暂停/恢复功能
- ❌ 错误提示（连接异常、消息解析失败等）
- ❌ 消息缓存（页面刷新后消息清空）
- ❌ 数据库持久化
- ❌ 性能优化（虚拟滚动、消息限流等）
- ❌ 搜索功能

### 6.2 其他
- 消息解析失败时忽略，不影响其他功能
- 连接异常时静默处理，无需提示

## 7. 后端改造需求（供参考）

当前后端需要实现以下逻辑：

### 7.1 浏览器客户端识别
在 WebSocket 连接建立时，检查请求头的 `User-Agent` 字段：
```javascript
const isBrowser = req.headers['user-agent']?.includes('Mozilla');
```

### 7.2 监控面板管理
- 维护一个监控面板客户端列表
- 向所有监控面板推送状态事件

### 7.3 消息推送逻辑
- 监控面板连接后，立即推送当前所有客户端的 `[CONNECT]` 事件
- 当有新客户端连接时，向所有监控面板推送 `[CONNECT]` 事件
- 当客户端断开时，向所有监控面板推送 `[DISCONNECT]` 事件
- 当客户端发送消息时，向所有监控面板推送 `[MESSAGE]` 事件

### 7.4 消息格式生成
```javascript
// 新连接
ws.send(`[CONNECT] ${clientId}-${ipAddress}`);

// 断开
ws.send(`[DISCONNECT] ${clientId}-${ipAddress}`);

// 消息
ws.send(`[MESSAGE] ${clientId}-${ipAddress}: ${message}`);
```

## 10. 开发建议

1. 使用 Vite 创建项目：`npm create vite@latest ws-monitor -- --template vanilla`
2. 使用原生 JavaScript 实现最简单的逻辑
3. 使用 CSS Flexbox 或 Grid 实现响应式布局
4. WebSocket 连接和消息处理逻辑封装为独立的模块
5. 使用浏览器原生 `history API` 实现路由（列表页 ↔ 详情页）
